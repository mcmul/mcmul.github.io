Slices

January 18th, 2021

Slices abstract over arrays. Let's start with the array.

Arrays

Go arrays are typed, have fixed lengths, do not need to be initialised, and default to zero values. The length forms part of its type:

var costs [3]int fmt.Println(costs) // [0, 0, 0] fmt.Println(reflect.TypeOf(costs)) // [3]int var revenue [4]int fmt.Println(revenue) // [0, 0, 0, 0] fmt.Println(reflect.TypeOf(revenue)) // [4]int

Arrays are value types. The entire contents of an array is copied when passed to a function; array pointers are used to improve performance.

myCosts = &costs myCosts[0] = 10 myCosts[2] = 30

Go supports array literals:

fixedCosts := [4]int{10, 20, 30, 40}

And has a little syntactic sugar to save counting:

varCosts := [...]int{1, 2, 3, 4, 5, 6, 7} fmt.Println(reflect.TypeOf(varCosts)) // [7]int

Slices

To reiterate, slices abstract over arrays. They have three properties:

  • A pointer to the first element of the underlying array
  • A length, which can be inspected with the built-in function len
  • A capacity, which can be inspected with the built-in function cap

The zero-value of a slice is nil:

var slice []string isNil := slice == nil // true

Initialise a slice with make:

newsletters := make([]string, 2) newsletters[0] = "Money Stuff by Matt Levine" newsletters[1] = "The Bluestocking by Helen Lewis"

Or use a slice literal:

newsletters := []string{ "Money Stuff by Matt Levine", "The Bluestocking by Helen Lewis", }

Unlike arrays, they have no fixed length:

finance := []string{"ft", "the economist"} // 2 general := []string{"bbc", "hn", "lrb", "nyrb"} // 4 fmt.Println(reflect.TypeOf(finance)) // []string fmt.Println(reflect.TypeOf(general)) // []string

Slices use zero-based indexing:

under3 := []int{0, 1, 2} // [0, 1, 2] under3[0:0] // [] under3[0:1] // [0] under3[0:2] // [0, 1] under3[0:3] // [0, 1, 2]

They can be indexed from zero through to the length of the underlying array.

The length of a slice is the number of items in the slice. Using the example above:

len(under3[0:0]) // 0: [] len(under3[0:1]) // 1: [0] len(under3[0:2]) // 2: [0, 1] len(under3[0:3]) // 3: [0, 1, 2]

The capacity of a slice is the length of the underlying array.

first := under3[0:1] // [0] len(first) // 1: [0] cap(first) // 3: [0, 1, 2]

The slice pointer is the memory address of the first element of the underlying array:

finance := []string{"ft", "the economist"} fmt.Printf("%p", finance) // 0xc0000a6020

Thus 0xc0000a6020 is the memory address of ft in the array type [2]string.

The slice pointer changes when the left-index changes. That is, slicing from [0:3] to [0:cap(slice)] has no effect on the slice pointer. But, a move from [0:3] to [2:3] does.

A few important points:

  • The length of a slice can only grow to its capacity
  • The left-index of a slice cannot be less than zero
  • The capacity of the slice decreases as the left-index increases

To illustrate, consider the below:

newsletters := []string{ "Matt Levine", "Helen Lewis", } fmt.Printf("%p\n", newsletters) // 0xc00000c080 newsletters = newsletters[1:] // remove Matt Levine len(newsletters) // 1 cap(newsletters) // 1 fmt.Printf("%p\n", newsletters) // 0xc00000c090

The re-slice of newsletters from [0:cap(newsletters)] to [1:] reduces the capacity from two to one. And, as I mentioned earlier, a slice cannot grow beyond its capacity: Matt Levine is lost forever (a tragic day!).

That said, this sort of isn't true. The built-in function append can, which:

appends elements to the end of a slice. If it has sufficient capacity, the destination is resliced to accommodate the new elements. If it does not, a new underlying array will be allocated. Append returns the updated slice. It is therefore necessary to store the result of append, often in the variable holding the slice itself

For example:

var pension = []string{"apple", "google"} pension = append(pension, "netflix", "tesla")

Go has another useful built-in function called copy, which:

copies elements from a source slice into a destination slice. (As a special case, it also will copy bytes from a string to a slice of bytes.) The source and destination may overlap. Copy returns the number of elements copied, which will be the minimum of len(src) and len(dst).

For example:

var savings = make([]string, 2) copy(savings, pension)

To finish, let me quote Andrew Gerrand from the excellent blog post Go Slices: usage and internals:

Arrays have their place, but they're a bit inflexible, so you don't see them too often in Go code. Slices, though, are everywhere. They build on arrays to provide great power and convenience.

... like the sort package: slices, everywhere!

Channels

July 28th, 2020

Channels are a special kind of synchronisation primitive in Go. Unlike the mutual exclusion lock or the concurrent map (two of many in the sync package), Channels are first-class types that need not be imported. Channels first originated in Tony Hoare's influential paper on Communicating Sequential Processes (CSP), which the designers of Go hold in high regard. They provide a mechanism for inter-process communication, which in Go terms means they let goroutines send and receive values. This is a simple but powerful idea, which Go promotes with the following proverb:

Don't communicate by sharing memory, share memory by communicating

This was long the mantra of Go, which is referenced in Effective Go and a blog post:

Traditional threading models (commonly used when writing Java, C++, and Python programs, for example) require the programmer to communicate between threads using shared memory. Typically, shared data structures are protected by locks, and threads will contend over those locks to access the data. In some cases, this is made easier by the use of thread-safe data structures such as Python's Queue.

Go's concurrency primitives - goroutines and channels - provide an elegant and distinct means of structuring concurrent software. (These concepts have an interesting history that begins with C. A. R. Hoare's Communicating Sequential Processes.) Instead of explicitly using locks to mediate access to shared data, Go encourages the use of channels to pass references to data between goroutines. This approach ensures that only one goroutine has access to the data at a given time.

With that, let's take a look at their usage.

Channel usage

The chan keyword denotes the channel type. Channels are typed, initialised with the make function, and their uninitialised value is nil. The following makes the composite type chan int, which is a channel that can send and receive integers:

c := make(chan int)

The operator <- denotes a send or receive.

To send, place it to the right of the channel:

c <- "I'm a send"

And to receive, place it to the left of the channel:

v := <-c

Channels have a capacity:

c := make(chan string, 10) // capacity of 10 c := make(chan int, 0) // equivalent to make(chan int)

A capacity of 0 is unbuffered; a capacity of >0 is buffered.

Sends on an unbuffered channel block until a receiver receives; receives on an unbuffered channel block until a sender sends.

c := make(chan int) // capacity of 0; unbuffered c <- 10 // blocks

Sends on a buffered channel block if the buffer is full; receives on a buffered channel block if the buffer is empty.

c := make(chan int, 1) // capacity of 1 c <- 10 // does not block fmt.Println(<-c) // prints the received value

Functions with a channel parameter can specify whether the channel is send-only or receive-only:

func main() { c1 := make(chan int) c2 := make(chan int) go f(c1, c2) // blocks without the go keyword c1 <- 10 fmt.Println(<-c2) } func f(receiveOnly <-chan int, sendOnly chan<- int) { sendOnly <- <-receiveOnly }

f receives from the receive-only channel, and then sends the value to the send-only channel. Note the comment on the invocation of f: the channel is unbuffered, which means it would block if it wasn't spawned in a new goroutine.

Channels may also be closed to indicate no more values will be sent. Closed channel sends panic; closed channel receives return the zero-value.

c := make(chan string) close(c) fmt.Println(<-c) // prints the zero-value: "" c <- "test" // panics

The receive operator can also return a boolean value to indicate whether the value was produced by a successful send operation:

c := make(chan string, 1) c <- "test" close(c) v1, ok := <-c fmt.Println(v1, ok) // prints "test" + "true" v2, ok := <-c fmt.Println(v2, ok) // prints "" + "false"

Channels can also be ranged over:

c := make(chan int, 2) go func() { for i := 0; i < 3; i++ { c <- i time.Sleep(time.Duration(i) * time.Second) } close(c) }() for v := range c { fmt.Println(v) }

The iteration stops when the channel is closed; the range deadlocks if the channel isn't closed and no more values are sent.

The composition of channels are somewhat analogous to UNIX pipelines: the output of one channel can be piped into the input of another. The official Go Blog has an excellent pipelines post. The example below creates a data pipeline with two stages: the first stage filters valid integers out of a list of strings, and the second stage squares them:

func main() { input := []string{"1", "2", "a", "b", "3", "2", "3"} for squared := range squarer(filter(input...)) { fmt.Println(squared) } } func filter(input ...string) <-chan int { c := make(chan int) go func() { for _, s := range input { i, err := strconv.Atoi(s) if err != nil { continue } c <- i } close(c) }() return c } func squarer(input <-chan int) <-chan int { c := make(chan int) go func() { for i := range input { c <- i * i } close(c) }() return c }

Selecting channels

Consider an application that's waiting on two channels. What happens if one is quiet and one is chatty? Won't it block? Or, at the very least, won't we be stuck waiting on the quiet channel when we could be dealing with the chatty channel?

Enter the select statement, which is like a switch statement for channels. Each case statement deals with a send or receive on a channel, and the default case handles the instances where no channel is ready.

Consider the example below. The select waits on two channels: the first receives the string BOOM every 5 seconds, and the second is a neat little function provided by the time package, which receives the current time every 1 second, and in this case just prints tick.

func main() { boomCh := boom(5) for { select { case v := <-boomCh: fmt.Println(v) case <-time.After(1 * time.Second): fmt.Println("tick") } } } func boom(interval int) <-chan string { c := make(chan string) go func() { for { time.Sleep(time.Duration(interval) * time.Second) c <- "BOOM" } }() return c }

And another example, which waits on a single channel and implements the default case:

func main() { c := make(chan string) go func() { for i := 0; i < 5; i++ { c <- strconv.Itoa(i) time.Sleep(time.Duration(i) * time.Second) } close(c) }() for { select { case v, ok := <-c: if !ok { fmt.Println("done") return } fmt.Println(v) default: fmt.Println("waiting") time.Sleep(1 * time.Second) } } }

And the final example, which waits on two channels: a send operation which is being received by a goroutine that stops receiving after 5; a receive on the neat little time function mentioned earlier, which in this case acts like a timeout (such that the program ends when the goroutine stops reading and two seconds have passed):

func main() { c := make(chan struct{}) go func() { for i := 0; i < 5; i++ { <-c time.Sleep(1 * time.Second) } }() for { select { case c <- struct{}{}: fmt.Println("sent") case <-time.After(2 * time.Second): fmt.Println("I'm done waiting") return } } }

To summarise, channels let goroutines communicate with each other. They're typed, can be buffered or unbuffered, and they don't need to be imported. The channel combined with the goroutine and select statement are what differentiates Go from other languages and enables its CSP inspired concurrency.

Goroutines

July 24th, 2020

Concurrency is just too easy in Go: put the go keyword in front of any function and voila, it runs concurrently. But why is it so easy and how does it work?

It begins with the goroutine. The main function of every Go program executes in a goroutine. They are independent lines of execution, similar to threads. However one key difference is the cost: goroutines are cheap. The default stack size of a POSIX thread is 2MB, goroutines start with 2048 bytes. Go uses resizable, bounded stacks, which the runtime grows and shrinks automatically. It is practical to run hundreds of thousands - even millions - of goroutines. Go does this through its M:N scheduler. Or, in other words, it multiplexes x goroutines over y OS threads that run on z processors (where z is a configurable number).

go doWork()

Essentially, a goroutine is a function that runs concurrently to other functions.

go worker(x, y, z)

Simply put the go keyword in front of any function call and Go will run the function in a new goroutine. All variables passed to the function are evaluated in the current goroutine. Thus variables x, y, and z in the example above are evaluated in the goroutine executing the go statement, and the worker function is executed in the new goroutine.

func main() { go func() { fmt.Println("hello, world") }() }

Anonymous functions can be spawned in goroutines too. However note the example above is quite interesting: it doesn't actually print anything. Why?

The Go language specification:

Program execution begins by initializing the main package and then invoking the function main. When that function invocation returns, the program exits. It does not wait for other (non-main) goroutines to complete.

In other words, it doesn't print anything because the main goroutine died before the spawned goroutine ran. This highlights an important point: Go provides no guarantee on when goroutines will run — only that it will run, if given enough time.

Go follows the fork-join model of parallelism: parent goroutines fork child goroutines, with the intention that the child runs in parallel to the parent. Then at some point in the future — usually when the child is finished processing — the child joins back up with the parent (the join point).

c := make(chan int) go worker(c) // fork <-c // join

The main goroutine (g1) forks the function worker in a new goroutine (g2). g2 joins back up with g1 when g1 receives on the channel.

func main() { wg := sync.WaitGroup{} wg.Add(1) go hello(wg) // fork wg.Wait() // join } func hello(wg sync.WaitGroup) { fmt.Println("hello, world") wg.Done() }

Perhaps a little more contrived: the main goroutine (g1) forks the function hello in a new goroutine (g2). g2 joins back up with g1 when g1 waits for the WaitGroup to finish (i.e. when the counter is back to zero).

Programmers should be mindful of when and where goroutines start (are forked) and finish (the join points). And, most importantly, know exactly when goroutines block (especially the main thread).

The words concurrency and parallelism were referenced in this post. Rob Pike has an excellent slide deck on the difference. He summarises it on slide 6:

Concurrency is the composition of independently executing computations.

Concurrency is a way to structure software, particularly as a way to write clean code that interacts well with the real world.

And then expands on slide 7:

Concurrency is not parallelism, although it enables parallelism.

If you have only one processor, your program can still be concurrent but it cannot be parallel.

On the other hand, a well-written concurrent program might run efficiently in parallel on a multiprocessor. That property could be important...

Or, as he put it even more simply in his talk:

Concurrency is about dealing with a lot of things at once, and parallelism is about doing a lot of things at once.

To summarise, goroutines are like cheap, lightweight threads. They have their own stack that grows and shrinks automatically; the Go runtime manages their distribution over n OS threads; one should be mindful of exactly when they start, when they finish, and where they block. Essentially, they allow the programmer to write concurrent programs. Get this right, and Go gives you the parallelism for free.
